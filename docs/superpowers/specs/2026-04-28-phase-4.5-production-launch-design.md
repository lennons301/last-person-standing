# Phase 4.5 — Production Launch Foundation

**Date:** 2026-04-28
**Status:** Approved
**Scope:** Take last-person-standing from "merges dormant to `main`" to "live on Vercel with CI/CD, secrets, monitoring, and the cron pipeline running" before WC kickoff (2026-06-11).

---

## Context

Phases 4a–4c5 shipped the full feature set, but every PR was merged with no production deployment. There is no Vercel project, no Doppler project, no Neon prod database, and no QStash provisioning. The existing `.github/workflows/live-scores.yml` cron has been failing every minute since 2026-04-23 because its repo secrets aren't set — this has been an intentional canary; 4.5 is when it goes green.

This is a **terminal sub-phase**. After 4.5, all subsequent phases ride CI/CD via Vercel's GitHub integration. The pattern mirrors `~/code/moontide`'s 2026-04-14 infrastructure spec, with adaptations for our extra dependencies (QStash, GitHub Actions cron, real Postgres in CI).

## Architecture

```
GitHub (lennons301/last-person-standing)
  ├── push to main      ──▶ Vercel Production deploy (region: lhr1)
  ├── PR opened         ──▶ Vercel Preview deploy (per-PR URL)
  └── CI workflow gates merges (lint/typecheck/test/build — already exists)

Doppler (project: last-person-standing)
  ├── dev               ──▶ local .env.local (manual, never synced)
  ├── stg               ──▶ Vercel Preview env (auto-sync via Doppler-Vercel integration)
  └── prd               ──▶ Vercel Production env (auto-sync via Doppler-Vercel integration)

Neon (project: last-person-standing, region: London/eu-west-2)
  ├── main branch       ──▶ DATABASE_URL in Doppler prd
  └── stg branch        ──▶ DATABASE_URL in Doppler stg

Upstash QStash
  ├── project last-person-standing-prd   ──▶ keys in Doppler prd
  └── project last-person-standing-stg   ──▶ keys in Doppler stg

GitHub Actions
  ├── ci.yml (existing) — gates merges
  ├── live-scores.yml (existing) — every-minute cron, needs CRON_SECRET + VERCEL_PROD_URL repo secrets
  └── migrate.yml (new) — runs Drizzle migrations on push to main, gated on CI
```

Single PR/branch (`feature/phase-4.5-production-launch`). No decomposition into sub-PRs. Internal organisation: three blocks — Provision → Deploy → Verify.

### B2. GitHub repo secrets

In Settings → Secrets and variables → Actions, set:

| Secret | Value | Used by |
|---|---|---|
| `CRON_SECRET` | Same value as Doppler `prd.CRON_SECRET` | `live-scores.yml` |
| `VERCEL_PROD_URL` | Set in Block C step 4 (after first deploy) | `live-scores.yml` |
| `PROD_DATABASE_URL` | Same value as Doppler `prd.DATABASE_URL` | `migrate.yml` |

`PROD_DATABASE_URL` duplicates Doppler intentionally — GitHub Actions can't read Doppler without a service token, and a repo secret is simpler than wiring `doppler-cli` into the migrate workflow. The trade-off: one extra place to update during a credential rotation. Acceptable for now; revisit if rotation cadence increases.

### B4. Code changes

#### B4a. `vercel.json` (new file)

```json

  "regions": ["lhr1"]

```

No Vercel Cron entries — `live-scores` runs as a GitHub Action; deadline jobs run via QStash. Vercel auto-detects Next.js; `vercel.json` only carries the region pin.

#### B4b. `.github/workflows/migrate.yml` (new file)

Runs Drizzle migrations on `main` after CI passes. Uses `workflow_run` trigger (not the simple-form `needs:` — that doesn't work across workflows).

```yaml
name: Migrate prod DB

on:
  workflow_run:
    workflows: [CI]
    branches: [main]
    types: [completed]

jobs:
  migrate:
    if: ${{ github.event.workflow_run.conclusion == 'success' }}
    runs-on: ubuntu-latest
    env:
      DATABASE_URL: ${{ secrets.PROD_DATABASE_URL }}
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.workflow_run.head_sha }}
      - uses: pnpm/action-setup@v4
        with:
          version: 9
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm exec drizzle-kit migrate
```

The `if:` guard ensures we only migrate when CI succeeded. The checkout pins to the head SHA of the CI run (not `main` HEAD) so we migrate exactly the commit that passed CI, even if subsequent merges have happened.

## Block C — Deploy + Verify

### C1. Deploy sequence

Strict order — `BETTER_AUTH_URL` and `VERCEL_PROD_URL` are chicken-and-egg with the first deploy.

1. With Block A and B in place, push `feature/phase-4.5-production-launch` → open PR → CI green → merge to `main`.
2. Vercel auto-deploys `main`. Capture the production URL from the Vercel dashboard (e.g., `https://last-person-standing.vercel.app` or `https://last-person-standing-lennons301s-projects.vercel.app`).
3. Update `BETTER_AUTH_URL` in Doppler `prd` to the captured URL. Doppler resyncs to Vercel; Vercel rebuilds production. Wait for the rebuild to complete.
4. Set GitHub repo secret `VERCEL_PROD_URL` to the captured URL.
5. From your laptop: run migrations against prod once.
   ```bash
   DATABASE_URL='<prod URL from Doppler prd>' pnpm exec drizzle-kit migrate
   ```
6. **Audit `scripts/seed.ts`** before running it against prod. Goal: insert WC competition + fixtures + teams (reference data) but NOT dev test users / sample games. If `seed.ts` mixes both, extract a `scripts/seed-prod.ts` containing only reference data, or perform reference inserts manually via SQL. Then either:
   ```bash
   DATABASE_URL='<prod URL>' pnpm tsx scripts/seed-prod.ts
   ```
   or apply the reference SQL manually.

### C2. Verification

- `https://<prod>.vercel.app/login` loads and renders.
- Sign up real account → log in → create a game from WC competition → make a pick → pick saves → dashboard shows it.
- `gh run list --workflow=live-scores.yml` shows green runs in the minute after `VERCEL_PROD_URL` + `CRON_SECRET` are set. (Hard criterion from `project_phase_4_5_must_haves.md`.)

If WC fixtures are too far out for a real test, send a fake QStash message via Upstash dashboard targeting `/api/qstash/handler` and watch the Vercel function log.
- Branch protection works: open a dummy PR with a deliberately failing test; confirm "Merge" is blocked until CI passes; close the dummy PR.

## Why this order

Block A (provision) must complete before Block B (Vercel project) because the Doppler-Vercel integration needs configs to exist before mapping them. Block B must complete before Block C (deploy) because the deploy reads env vars from Doppler via Vercel. Within Block C, the steps are strictly ordered because `BETTER_AUTH_URL`, `VERCEL_PROD_URL`, and migrations all depend on knowing the prod URL or having the schema in place.

## Risks

- **Doppler-Vercel sync lag.** If sync hasn't completed when the first deploy runs, the build will fail with missing env vars. Mitigation: Block B's acceptance includes "integration shows green sync status" before merging.
- **Seed script writes dev data to prod.** Mitigation: explicit audit step in C1.6.
- **`live-scores.yml` keeps failing after secrets are set.** Possible if the secrets contain whitespace or the URL is missing the protocol. Mitigation: verification step explicitly checks `gh run list`.
- **QStash callback can't reach the Vercel deployment.** Possible if `VERCEL_URL` env var (used as callback base) isn't set right or auth fails on the QStash handler route. Mitigation: explicit smoke test in C2.
- **Post-merge schema race.** `migrate.yml` runs in parallel with the Vercel deploy, so new code can be live for ~1 minute before its required schema changes apply. Backwards-compatible (additive) migrations are safe; destructive or incompatible migrations are not. Operational practice: for any non-backwards-compatible schema change, the developer applies the migration manually against prod *before* merging the PR. The automation covers the additive 99% case.
